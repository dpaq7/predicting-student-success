# Data Science Project Template - Complete Guide

## 📌 Overview

A modern template for data science projects with all the necessary tools for experiment, development, testing, and deployment. From notebooks to production.

**Documentation**: [https://joserzapata.github.io/data-science-project-template/](https://joserzapata.github.io/data-science-project-template/)  
**Source Code**: [https://github.com/JoseRZapata/data-science-project-template](https://github.com/JoseRZapata/data-science-project-template)

## ✨ Core Features

- **Dependency management** with [UV](https://docs.astral.sh/uv/) (replaces pip, poetry, pyenv, virtualenv)
- **Virtual environment management** with [UV](https://docs.astral.sh/uv/)
- **Linting** with [pre-commit](https://pre-commit.com/) and [Ruff](https://docs.astral.sh/ruff/)
- **Continuous integration** with [GitHub Actions](https://github.com/features/actions)
- **Documentation** with [mkdocs](https://www.mkdocs.org/) and [mkdocstrings](https://mkdocstrings.github.io/) using [mkdocs-material](https://github.com/squidfunk/mkdocs-material) theme
- **Automated dependency updates** with [Dependabot](https://github.com/dependabot/dependabot-core)
- **Code formatting** with [Ruff](https://docs.astral.sh/ruff/)
- **Import sorting** with [Ruff](https://docs.astral.sh/ruff/) using isort rule
- **Testing** with [pytest](https://docs.pytest.org/en/latest/)
- **Code coverage** with [Coverage.py](https://coverage.readthedocs.io/)
- **Coverage reporting** with [Codecov](https://codecov.io/)
- **Static type-checking** with [mypy](http://mypy-lang.org/)
- **Security audit** with [Ruff](https://docs.astral.sh/ruff/) using bandit rule
- **Manage project labels** with [GitHub Labeler](https://github.com/marketplace/actions/github-labeler)
- **Project configuration** with [Hydra](https://hydra.cc/)

## 📁 Creating a New Project

### 👍 Prerequisites & Recommendations

#### UV Installation (Recommended Python Management Tool)

UV is an extremely fast tool that replaces pip, pip-tools, Pipx, Poetry, Pyenv, twine, virtualenv, and more.

**Install UV:**
```bash
# On macOS and Linux
curl -LsSf https://astral.sh/uv/install.sh | sh

# On Windows
powershell -c "irm https://astral.sh/uv/install.ps1 | iex"

# Or via pip
pip install uv
```

### 🍪 Project Creation Methods

#### Method 1: Via Cruft (Recommended) 🥇

Cruft enables automatic template synchronization, keeping your project updated with template improvements.

```bash
# Install cruft in an isolated environment using uv
uv tool install cruft

# Or install with pip
pip install --user cruft

# Create new project
cruft create https://github.com/JoseRZapata/data-science-project-template

# Then inside the project folder, initialize git and uv environment
make init_git
make init_env
```

#### Method 2: Via Cookiecutter

```bash
# Install cookiecutter in an isolated environment
uv tool install cookiecutter

# Or install with pip
pip install --user cookiecutter

# Create new project
cookiecutter gh:JoseRZapata/data-science-project-template
```

**Note**: Cookiecutter uses `gh:` as shorthand for `https://github.com/`

### 🔗 Linking an Existing Project

If the project was originally installed via Cookiecutter, first link it with Cruft:

```bash
cruft link https://github.com/JoseRZapata/data-science-project-template
```

Then you can update the template:
```bash
cruft update
```

## 🔨 Makefile Commands

The template includes a self-documenting Makefile. Type `make` in the terminal to see available commands:

| Target | Description |
|--------|-------------|
| `check` | Run code quality tools with pre-commit hooks |
| `docs_test` | Test if documentation can be built without warnings or errors |
| `docs_view` | Build and serve the documentation |
| `init_env` | Install dependencies with uv and activate env |
| `init_git` | Initialize git repository |
| `install_data_libs` | Install pandas, scikit-learn, Jupyter, seaborn |
| `pre-commit_update` | Update pre-commit hooks |
| `test` | Test the code with pytest and coverage |

## 📂 Data Organization

The template follows a layered approach to data organization:

1. **01_raw**: Original, immutable data dump
2. **02_intermediate**: Cleaned and typed data
3. **03_primary**: Domain model data
4. **04_feature**: Engineered features for modeling
5. **05_model_input**: Final datasets for training (master tables)
6. **06_models**: Serialized trained models
7. **07_model_output**: Predictions and model outputs
8. **08_reporting**: Final reports and visualizations

## 📓 Notebook Organization

Notebooks are organized by workflow stage:

1. **1-data**: Data extraction and initial cleaning
2. **2-exploration**: Exploratory Data Analysis (EDA)
3. **3-analysis**: Statistical analysis and hypothesis testing
4. **4-feat_eng**: Feature engineering and selection
5. **5-models**: Model training and hyperparameter tuning
6. **6-interpretation**: Model interpretation and explanation
7. **7-deploy**: Deployment preparation and strategies
8. **8-reports**: Final reports and storytelling

## 🎯 Source Code Organization

The `src/` directory follows a modular structure:

- **data/**: Data processing modules
- **model/**: Model training and evaluation
- **inference/**: Prediction and serving
- **pipelines/**: End-to-end pipeline orchestration
  - Feature pipeline
  - Training pipeline
  - Inference pipeline

## 🏃 Getting Started Workflow

1. **Create the project** using Cruft or Cookiecutter
2. **Initialize the environment**:
   ```bash
   cd your-project-name
   make init_git  # Initialize git repository
   make init_env  # Setup Python environment with uv
   ```
3. **Install data science libraries**:
   ```bash
   make install_data_libs  # Installs pandas, scikit-learn, Jupyter, seaborn
   ```
4. **Start developing**:
   - Place raw data in `data/01_raw/`
   - Create notebooks in appropriate `notebooks/` subdirectories
   - Develop reusable code in `src/`
   - Write tests in `tests/`
5. **Run quality checks**:
   ```bash
   make check  # Run all pre-commit hooks
   make test   # Run tests with coverage
   ```
6. **Build documentation**:
   ```bash
   make docs_view  # Build and serve documentation locally
   ```

## 🎓 Why This Structure?

### Layered Data Architecture
The numbered data folders follow the principle of layered thinking in data engineering, where each layer represents a different stage of data refinement, making it easy to track data lineage and maintain reproducibility.

### Separation of Concerns
- **Notebooks** for experimentation and exploration
- **Source code** for production-ready, reusable components
- **Tests** for ensuring code reliability
- **Documentation** for knowledge sharing

### Automation First
Every repetitive task is automated through Make commands, pre-commit hooks, or CI/CD pipelines, allowing data scientists to focus on analysis and modeling rather than infrastructure.

### Modern Tooling
Uses the latest and fastest tools (UV, Ruff) that significantly improve developer experience through speed and unified interfaces.

## 💡 Tips for Success

1. **Start with the template**: Don't modify the structure until you've used it for a project
2. **Use the Makefile**: Familiarize yourself with all available commands
3. **Write tests early**: Add tests as you develop new functionality
4. **Document as you go**: Update docstrings and documentation continuously
5. **Leverage pre-commit**: Let automation handle code quality
6. **Regular updates**: If using Cruft, update the template periodically
7. **Follow the data flow**: Process data through the numbered folders sequentially
8. **Modularize code**: Move notebook code to `src/` once it's stable
